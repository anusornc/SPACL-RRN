\begin{abstract}
We present SPACL (Speculative Parallel Tableaux with Adaptive Conflict Learning), an open-source ALC/SHOIQ reasoner that combines speculative parallelism with conflict-driven learning. The reference implementation and repository are named \textit{Tableauxx}; throughout this manuscript, we use \textit{SPACL} for the algorithm/system and \textit{Tableauxx} only when referring to the software artifact. Relative to mainstream open OWL2 DL pipelines discussed in Section~\ref{sec:related}, SPACL emphasizes the combination of work-stealing speculative execution and reusable nogood information in one implementation path.

Our key contributions are: (1) a speculative parallel tableaux framework with work-stealing scheduling; in the current prototype this is realized via branch-constrained work items evaluated in parallel; (2) an adaptive threshold mechanism that selects sequential or parallel execution based on estimated complexity, yielding 18--73$\times$ speedup over naive always-parallel execution on small ontologies; (3) thread-local nogood caching that keeps most repeated nogood lookups on the local hot path in dedicated inconsistent-workload ablations (82--95\% local cache-hit rates); (4) a global thread pool optimization that reduces overhead by 2.1$\times$; and (5) a Rust implementation with parser support for RDF/XML, OWL2 Functional, Manchester, and JSON-LD syntaxes. On synthetic scalability workloads, SPACL reaches up to 4.88$\times$ speedup at 10,000 classes while keeping overhead below $2\times$ on smaller ontologies.

In the profile-mixed Standard Workload Suite (7 ontologies; 3 clean runs), per-ontology medians among successful comparable runs place SPACL lowest on 4/7 ontologies; ELK (OWL EL profile) is lowest on the two deepest hierarchy files, and Pellet is lowest on \texttt{univ-bench.owl}. In the primary Large Head-to-Head benchmark (doid/go-basic/uberon; 3 clean runs), SPACL has the lowest median wall-clock time on all three tested ontologies among successful comparable runs under a common 900s timeout protocol. A suite-level failure-aware metric (capped expected wall-time, CEW) over comparable runs reports ELK lowest on Standard and SPACL lowest on Large. In an external OWL2Bench core-profile validation (DL/EL/QL/RL), three clean repeated runs show SPACL has the lowest median wall-clock time on 4/4 ontologies with 12/12 successful runs in this harness. Timeout policy is reported explicitly in all benchmark tables (Standard Workload Suite: 300s; Large Head-to-Head: 900s; OWL2Bench core profile suite: 300s). This improvement is driven by a structural RDF/XML parser and optimized materialization path that reuses the same ALC/SHOIQ semantic handlers as the baseline path; stage-level A/B runs show parse-only reductions of 96.96--97.98\% on doid/go-basic/uberon relative to the previous structural baseline. Head-to-head suites are reported as descriptive repeated-run wall-clock comparisons, not formal significance tests.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Description Logics (DLs) provide the formal basis of the Semantic Web, and OWL2 DL is the corresponding W3C standard ontology language \cite{owl2}. Tableaux methods are still widely used for OWL2 DL reasoning because they provide sound and complete procedures for expressive TBox and ABox tasks \cite{dlhandbook}. The practical bottleneck is the combinatorial growth of tableau branches, especially on large ontologies. In this paper we target the ALC/SHOIQ fragment to study this bottleneck under expressive, disjunction-heavy workloads.

Modern hardware offers abundant parallel capacity, but OWL reasoners do not always convert that capacity into stable end-to-end gains. Some strong parallel systems are integrated as external binaries in practical harnesses (e.g., Konclude) \cite{konclude}, which can limit instrumentation parity and failure diagnosis in cross-system experiments. Many widely used open implementations (e.g., Pellet/Openllet and HermiT) remain primarily sequential in their core exploration loop \cite{pellet,openllet,hermit}. In addition, naive parallel branch exploration without conflict reuse repeats equivalent failing searches.
These constraints also motivate our longer-term goal: a reasoning core for ontology transaction validation in distributed, permissioned application settings. This paper focuses on the reasoning and parsing foundations required for that deployment path.

\subsection{Research Questions}
\label{sec:questions}

This work addresses the following research questions:

\begin{enumerate}[RQ1:]
    \item Can speculative parallel tableaux with conflict-driven nogood learning improve OWL2 DL consistency checking performance?
    \item Can adaptive parallelism reduce overhead on small workloads while preserving scalability on larger workloads?
    \item For large RDF/XML ontologies, which stages dominate end-to-end runtime, and how much can structural parsing/materialization optimization reduce this bottleneck while retaining implementation-aligned semantic behavior?
\end{enumerate}

\subsection{Contributions}
\label{sec:contributions}

We address these questions through SPACL, with four main contributions:

\begin{enumerate}[C1:]
    \item \textbf{Speculative parallel tableaux framework with conflict learning}: SPACL combines work-stealing execution with nogood extraction and reuse. The current prototype realizes speculative exploration via branch-constrained work items generated from selected disjunctions, reducing redundant failure exploration in non-deterministic search (Sections \ref{sec:workstealing}, \ref{sec:nogood}).
    
    \item \textbf{Adaptive overhead control}: We integrate threshold-based strategy selection with a global thread pool, reducing small-workload parallel overhead while preserving speedup at larger scales (Sections \ref{sec:adaptive}, \ref{sec:threadpool}).
    
    \item \textbf{Structural RDF/XML pipeline with handler-equivalent materialization}: We implement a structural parsing/materialization path for the evaluated ALC/SHOIQ handling path and substantially reduce parser-dominated runtime on large ontologies (Section \ref{sec:structural-parser}).
    
    \item \textbf{Reproducible end-to-end evaluation}: We provide a containerized benchmark protocol and artifacts for stage-level analysis and multi-reasoner head-to-head comparison (Sections \ref{sec:benchmark-protocol}, \ref{sec:competitor-benchmarks}).
\end{enumerate}

In Rust, these design choices translate into measurable gains on our synthetic benchmark groups: up to 4.88$\times$ speedup at 10,000 classes, measured per-class processing below 1$\mu$s at the largest synthetic point, and under $2\times$ overhead on small ontologies. On the same synthetic scalability workloads, SPACL reaches 26.2 million measured tableau rule applications per second.

\section{Related Work}
\label{sec:related}

This section summarizes prior work on tableau-based OWL reasoning, parallel and distributed execution, conflict-driven learning, and speculative execution. The goal is to position SPACL against the strongest established techniques used in practical reasoners.

\subsection{Tableau-Based Reasoning}

Tableau calculi remain a central approach for expressive description logic reasoning. Practical systems such as Pellet and HermiT demonstrated that sound and complete reasoning can be delivered at scale when rule ordering, blocking, and caching are engineered carefully~\cite{pellet,hermit,motik2009hypertableau}. In particular, the hypertableau line of work reduces non-deterministic branching pressure compared with classical tableau expansion, and individual-reuse techniques reduce model blow-up in challenging ontologies~\cite{motik2009hypertableau}.

State-of-the-art systems also combine calculi. Konclude integrates tableau-style reasoning with additional optimization layers and has shown strong performance on large benchmark sets~\cite{konclude,song2013complete,glimm2014coupling}. For tractable fragments, consequence-based reasoners such as ELK provide excellent throughput, illustrating how calculus design strongly influences scalability and implementation trade-offs~\cite{elk}.

\subsection{Parallel Reasoning}

Parallelization has been studied at multiple granularities. Shared-memory classification frameworks distribute satisfiability and subsumption checks across workers and report substantial speedup on multicore systems~\cite{quan2017parallel,quan2019framework}. Work-stealing style scheduling further improves load balancing when ontology subproblems are uneven~\cite{workstealing,quan2017parallel}.

Beyond classification, parallel ABox reasoning has explored dynamic splitting of model-construction tasks with cache reuse strategies to improve throughput on assertion-heavy workloads~\cite{steigmiller2020parallelised}. Distributed materialization frameworks (for less expressive profiles) use Spark-based execution and data-partitioned pipelines to scale to very large datasets~\cite{liu2017spowl,gu2015cichlid,benitez2023nora}. These results confirm the scalability potential of parallel infrastructures, while also highlighting that end-to-end performance is highly sensitive to reasoning profile, task type, and synchronization overhead.

\subsection{Conflict-Driven Learning}

Conflict-driven learning is a mature optimization paradigm in SAT/CSP, where learned clauses/nogoods prevent repeated exploration of failing search regions~\cite{marques1999grasp,glorian2020nacre}. In description logic systems, related ideas appear in more localized forms, such as nogood-aware individual reuse in hypertableau procedures~\cite{motik2009hypertableau}.

Hybrid reasoners that couple tableau and saturation procedures also demonstrate the value of reusing derived information to reduce repeated work during classification~\cite{song2013complete,glimm2014coupling,konclude}. However, openly documented integrations of branch-level speculative parallel tableau exploration with adaptive, reusable nogood learning remain limited in mainstream open-source OWL2 DL pipelines.

\subsection{Speculative Execution}

Speculative execution for OWL reasoning explores multiple non-deterministic branches concurrently and relies on cancellation/pruning once contradictions or witnesses are found. Work-stealing provides a practical runtime foundation for this model by balancing branch workloads dynamically~\cite{workstealing,quan2017parallel}. Lock-free and low-contention engineering techniques are also critical for obtaining net gains rather than synchronization-bound slowdowns~\cite{elk,quan2019framework}.

The primary systems challenge is overhead control. On easy workloads, naive always-parallel execution can underperform sequential reasoning because scheduling and coordination costs dominate. This motivates adaptive policies that switch between sequential and parallel execution based on estimated branch complexity~\cite{quan2017parallel}. A second challenge is conflict sharing: learned nogoods should be reused aggressively enough to prune search, but not so aggressively that contention erodes parallel benefit~\cite{steigmiller2020parallelised}.

\subsection{Summary and Positioning}

Prior literature provides strong building blocks: optimized tableau/hypertableau calculi~\cite{motik2009hypertableau,hermit}, high-throughput parallel classification frameworks~\cite{quan2017parallel,quan2019framework}, distributed materialization engines~\cite{liu2017spowl,gu2015cichlid,benitez2023nora}, and mature nogood machinery in constraint solving~\cite{glorian2020nacre}. SPACL is positioned at their intersection: branch-level speculative parallelism, adaptive overhead control, and practical nogood reuse designed for expressive OWL reasoning workloads.
To make this positioning analytically precise, the next section formalizes the decision model, branching behavior, and correctness arguments that underlie the implementation and evaluation.
\section{Formal Framework}
\label{sec:formal}

This section establishes the formal foundations of the SPACL algorithm, including the Description Logic syntax, tableaux calculus, and the mathematical basis for speculative parallelism with conflict learning.

\subsection{Description Logic ALC}
\label{sec:dl-syntax}

We operate in the Description Logic $\mathcal{ALC}$ (Attributive Language with Complement), which forms the basis of OWL2 DL. Our implementation extends to $\mathcal{SHOIQ}$ (with transitive roles, role hierarchies, nominals, inverse roles, and qualified cardinality restrictions). The syntax is defined as follows:

\begin{definition}[ALC Syntax]
Let $\mathbf{C}$ be a set of concept names, $\mathbf{R}$ a set of role names, and $\mathbf{I}$ a set of individual names. \emph{Concepts} $C, D$ are defined by the grammar:
\begin{align}
C, D &::= A \mid \top \mid \bot \mid \neg C \mid C \sqcap D \mid C \sqcup D \mid \exists r.C \mid \forall r.C
\end{align}
where $A \in \mathbf{C}$ is an atomic concept and $r \in \mathbf{R}$ is a role.
\end{definition}

An \emph{ontology} $\mathcal{O} = (\mathcal{T}, \mathcal{A})$ consists of a TBox $\mathcal{T}$ (terminology) and ABox $\mathcal{A}$ (assertions):
\begin{align}
\mathcal{T} &::= \{C \sqsubseteq D\} \cup \{C \equiv D\} \\
\mathcal{A} &::= \{a : C\} \cup \{(a,b) : r\}
\end{align}

\subsection{Tableaux Calculus}
\label{sec:tableaux-calculus}

The tableaux algorithm determines consistency by constructing a \emph{completion graph} $\mathcal{G} = (V, E, \mathcal{L})$ where:
\begin{itemize}
    \item $V$ is a set of nodes (individuals)
    \item $E \subseteq V \times \mathbf{R} \times V$ are labeled edges (role assertions)
    \item $\mathcal{L}: V \rightarrow 2^{\mathcal{C}}$ maps nodes to sets of concepts
\end{itemize}

\begin{definition}[Clash]
A node $x \in V$ contains a \emph{clash} if either:
\begin{enumerate}
    \item $\{C, \neg C\} \subseteq \mathcal{L}(x)$ for some concept $C$, or
    \item $\bot \in \mathcal{L}(x)$
\end{enumerate}
\end{definition}

The tableaux algorithm applies \emph{expansion rules} until either a clash-free complete graph is found (consistent) or all branches lead to clashes (inconsistent).
Table~\ref{tab:expansion-rules} summarizes the core $\mathcal{ALC}$ expansion rules used in our formal presentation.

\begin{table}[htbp]
\centering
\caption{Tableaux Expansion Rules for $\mathcal{ALC}$}
\label{tab:expansion-rules}
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Rule} & \textbf{Premise} & \textbf{Action} \\
\midrule
$(\sqcap)$ & $x : C \sqcap D \in \mathcal{L}(x)$ & $\mathcal{L}(x) := \mathcal{L}(x) \cup \{C, D\}$ \\
$(\sqcup)$ & $x : C \sqcup D \in \mathcal{L}(x)$ & Create branches: $\mathcal{L}_1(x) = \mathcal{L}(x) \cup \{C\}$, \\
& & $\mathcal{L}_2(x) = \mathcal{L}(x) \cup \{D\}$ \\
$(\exists)$ & $x : \exists r.C \in \mathcal{L}(x)$ & Add new $y$, $(x,y):r$, $y:C$ \\
$(\forall)$ & $x : \forall r.C \in \mathcal{L}(x)$, $(x,y):r$ & $\mathcal{L}(y) := \mathcal{L}(y) \cup \{C\}$ \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Speculative Parallelism Model}
\label{sec:parallel-model}

Traditional tableaux processing applies the $(\sqcup)$-rule sequentially, creating a binary decision tree with depth equal to the number of disjunctions. SPACL introduces \emph{speculative parallelism}:

\begin{definition}[Speculative Branch]
Given a disjunction $C \sqcup D$ at node $x$, a \emph{speculative branch} is a pair of subgraphs $(\mathcal{G}_1, \mathcal{G}_2)$ where:
\begin{align}
\mathcal{G}_1 &= (V, E, \mathcal{L}[x \mapsto \mathcal{L}(x) \cup \{C\}]) \\
\mathcal{G}_2 &= (V, E, \mathcal{L}[x \mapsto \mathcal{L}(x) \cup \{D\}])
\end{align}
Both branches are explored concurrently by different workers.
\end{definition}

\noindent\textbf{Heuristic speedup model.}
Let $T_{seq}$ be sequential runtime and $T_{par}$ speculative-parallel runtime with $p$ workers. A practical lower-bound model is:
\begin{align}
T_{par} \gtrsim \frac{T_{seq}}{p} \cdot (1 + \alpha \cdot o_{sync})
\end{align}
where $\alpha$ is the synchronized-work fraction and $o_{sync}$ is synchronization overhead. We use this expression as an intuition aid for overhead analysis, not as a formal theorem.

\subsection{Nogood Learning}
\label{sec:nogood-formal}

\begin{definition}[Nogood]
A \emph{nogood} is a set of class expressions $\mathcal{N}=\{C_1,\ldots,C_n\}$ that is jointly unsatisfiable under conjunction in the current branch context, i.e., $\bigcap_{i=1}^{n} C_i \sqsubseteq \bot$.
\end{definition}

\begin{definition}[Nogood Subsumption]
A nogood $\mathcal{N}$ \emph{subsumes} a branch with constraints $\Gamma$ if:
\begin{align}
\mathcal{N} \subseteq \Gamma
\end{align}
If $\mathcal{N}$ subsumes $\Gamma$, the branch can be immediately pruned.
\end{definition}

\begin{algorithm}[htbp]
\caption{Verified Nogood Extraction (Pruning-Safe)}
\label{alg:nogood}
\begin{algorithmic}[1]
\Require Contradictory branch constraints $\Gamma_b$, ontology $\mathcal{O}$
\Ensure Verified nogood $\mathcal{N}$ for pruning, or $\emptyset$
\State $\mathcal{N} \gets \Gamma_b$
\For{each constraint expression $e \in \Gamma_b$}
    \If{$\mathcal{O} \cup (\mathcal{N}\setminus\{e\}) \models \bot$}
        \State $\mathcal{N} \gets \mathcal{N}\setminus\{e\}$ \Comment{Greedy core minimization}
    \EndIf
\EndFor
\If{$\mathcal{O} \cup \mathcal{N} \models \bot$}
    \State \Return $\mathcal{N}$
\Else
    \State \Return $\emptyset$ \Comment{No pruning-authorized nogood}
\EndIf
\end{algorithmic}
\end{algorithm}

\subsection{Adaptive Threshold}
\label{sec:adaptive-formal}

The adaptive threshold mechanism uses two implementation-aligned gates to decide between sequential and parallel processing.

\begin{definition}[Implementation Cost Model]
For expression-level scoring, the prototype uses:
\begin{align}
\text{cost}(\text{ObjectUnionOf}(ops)) &= 50 \times |ops| + \sum \text{cost}(op) \\
\text{cost}(\text{ObjectIntersectionOf}(ops)) &= 25 \times |ops| + \sum \text{cost}(op) \\
\text{cost}(\text{ObjectComplementOf}(e)) &= 15 + \text{cost}(e) \\
\text{cost}(\text{atomic}) &= 10
\end{align}
\end{definition}

\begin{definition}[Two-Gate Parallel Enable Rule]
Let $B(\mathcal{O})$ be branch-count estimate:
\begin{align}
B(\mathcal{O}) = \max\left(1,\ |\mathcal{D}| + 2\times |\text{DisjointAxioms}| + \frac{|\text{Classes}|}{10}\right)
\end{align}
and let $C(\mathcal{O}) = \sum_{d \in \mathcal{D}} \text{cost}(d)$ over detected disjunctions. Parallel mode is enabled only when:
\begin{align}
B(\mathcal{O}) \ge \tau \quad \land \quad C(\mathcal{O}) \ge \theta \cdot p
\end{align}
where $\tau=2000$, $\theta=500\mu s$, and $p$ is worker count.
\end{definition}
This formalization mirrors the implemented gates reported in Sections \ref{sec:adaptive} and \ref{sec:adaptive-opt}. The coefficients (50, 25, 15, 10) are implementation scoring weights calibrated from ablation sweeps over synthetic disjunctive workloads and representative warm-up real-world ontologies; they are used only for relative gating and are not description-logic complexity constants or predictive runtime coefficients.

\noindent\textbf{Heuristic gate inequality.}
For estimated disjunction cost $D$ and parallel overhead $o_{par}$ with $p$ workers, a useful crossover rule is:
\begin{align}
D > \frac{p \cdot o_{par}}{1 - \alpha}
\end{align}
where $\alpha$ is an empirical nogood-hit proxy. This inequality motivates threshold tuning, but it is not presented as a formal proof obligation.

\subsection{Correctness (Idealized Model)}
\label{sec:correctness}

The theorem statements in this subsection refer to an idealized SPACL calculus with fair exhaustive exploration of generated disjunctive branches and a complete underlying sequential tableaux reference procedure. The current prototype applies engineering bounds (selected disjunction scope, bounded speculative depth, and timeout limits), so these results are calculus-level properties under those assumptions, not empirical guarantees for every configured run of the current implementation. Evaluation claims in Section~\ref{sec:evaluation} therefore remain observational rather than completeness claims.

\begin{theorem}[Idealized Soundness]
If SPACL returns \emph{consistent} for ontology $\mathcal{O}$, then $\mathcal{O}$ is satisfiable.
\end{theorem}

\begin{proof}
SPACL explores branches speculatively but applies the same expansion rules as sequential tableaux. Each worker maintains a valid completion graph. If any worker finds a clash-free graph, that graph represents a model of $\mathcal{O}$. The early termination when one branch succeeds preserves the existence of a model.
\end{proof}

\begin{theorem}[Idealized Conditional Completeness]
Assume fair exhaustive exploration of the generated disjunctive branches and a complete sequential tableaux base procedure for the target logic fragment. If ontology $\mathcal{O}$ is consistent, then the idealized SPACL calculus will eventually return \emph{consistent}.
\end{theorem}

\begin{proof}
Under the stated assumptions, speculative parallelism may reorder branch exploration but does not remove any satisfiable branch from eventual consideration. Nogood pruning removes only branches already verified unsatisfiable. Because the underlying sequential tableaux procedure is complete and the idealized SPACL schedule is assumed to explore generated branches fairly and exhaustively, any model reachable in the sequential calculus remains reachable in the idealized parallel calculus. Hence a consistent ontology eventually yields a clash-free branch.
\end{proof}

    \item \textbf{Speculative Work-Stealing}: Branches are dynamically distributed across worker threads using a work-stealing deque \cite{workstealing}. Each worker maintains a local queue, stealing from others when idle.
    
    \item \textbf{Nogood Learning}: Contradictions are analyzed to extract sound conflict summaries (nogoods) from branch constraints. These are cached and checked before branch expansion.
    
    \item \textbf{Adaptive Thresholding}: Problem complexity is estimated from axiom structure. Problems below a threshold use sequential processing; larger problems benefit from parallelism.
\end{enumerate}

Algorithm \ref{alg:spacl} presents the main consistency checking procedure.

\begin{algorithm}[htbp]
\caption{SPACL Consistency Check}
\label{alg:spacl}
\begin{algorithmic}[1]
\Require Ontology $O$, Configuration $C$
\Ensure \textbf{true} if $O$ is consistent, \textbf{false} otherwise

\State $branches \gets \textsc{EstimateBranchCount}(O)$
\If{$branches < C.\text{parallel\_threshold}$} \label{line:threshold}
    \State \Return $\textsc{SequentialConsistencyCheck}(O)$ \label{line:sequential}
\EndIf

\State $Q \gets \textsc{InitializeWorkQueue}(O)$ \label{line:init}
\State $N \gets \textsc{InitializeNogoodDatabase}()$
\State $W[1..C.\text{num\_workers}] \gets \textsc{StartWorkers}(Q, N, C)$

\While{$\textsc{NotTerminated}()$}
    \State $result \gets \textsc{CollectResults}(C.\text{timeout})$
    \If{$result = \text{SAT}$} \Return \textbf{true}
    \ElsIf{$result = \text{UNSAT}$} \Return \textbf{false}
    \ElsIf{$result = \text{TIMEOUT}$} \Return \textbf{UNKNOWN}
    \EndIf
\EndWhile
\end{algorithmic}
\end{algorithm}

\subsection{Speculative Work-Stealing}
\label{sec:workstealing}

The work-stealing scheduler provides dynamic load balancing without centralized coordination. Each worker thread maintains:

\begin{itemize}
    \item \textbf{Local Queue}: Double-ended queue (deque) of pending WorkItems
    \item \textbf{Stealer Handle}: Allows other threads to steal work from the tail
    \item \textbf{Local Statistics}: Branch counts, nogood hits, cache metrics
\end{itemize}

When parallel mode is enabled, the prototype pushes a root branch-constrained WorkItem and expands it recursively over the selected disjunction list. Each split generates one child WorkItem per disjunct operand, carrying accumulated branch constraints and an incremented split index; expansion stops when the disjunction list is exhausted or the speculative-depth bound is reached. Idle workers steal from the tail of busy workers' queues using a lock-free deque-based work-stealing strategy \cite{workstealing}.

\subsection{Nogood Learning and Caching}
\label{sec:nogood}

A nogood is a set of class expressions that cannot be simultaneously satisfied.

\begin{definition}[Nogood]
\label{def:nogood}
A nogood is a set of class expressions $\{C_i\}_{i=1}^{n}$ such that their conjunction is unsatisfiable:
\begin{equation}
\bigsqcap_{i=1}^{n} C_i \sqsubseteq \bot
\end{equation}
\end{definition}

When a contradiction is detected, the prototype learns a branch-level nogood from the active branch constraints. It first validates branch inconsistency using the same consistency checker, then applies greedy constraint minimization (remove-one-if-still-inconsistent). Only non-empty verified minimized cores are admitted to the nogood store; if minimization yields no verified core, no new nogood is learned for that contradiction. Before expanding a node, its constraint set is checked against known nogoods.
Algorithm~\ref{alg:nogood} summarizes this verified extraction flow.

\subsubsection{Thread-Local Caching}
\label{sec:cache}

Each worker maintains three levels of nogood storage:

\begin{enumerate}
    \item \textbf{Local Nogoods}: Discovered by this worker (no locks needed)
    \item \textbf{Cached Global Nogoods}: Copy of shared database (periodically synced)
    \item \textbf{Global Database}: Shared across all workers (synchronized access)
\end{enumerate}

The cache check algorithm proceeds as follows:

\begin{enumerate}
    \item Check local nogoods (fastest, no synchronization)
    \item Check cached global nogoods (fast, no synchronization)
    \item If cache miss and sync interval reached:
    \begin{enumerate}
        \item Acquire read lock on global database
        \item Update cache with new nogoods
        \item Release lock
        \item Re-check with updated cache
    \end{enumerate}
    \item If still no hit, proceed with expansion
\end{enumerate}

Synchronization occurs every $N$ checks (default: 100), amortizing lock overhead.

\subsubsection{Correctness of Nogood Learning}
\label{sec:nogood-correctness}

We establish the soundness of SPACL's nogood learning mechanism.

\begin{definition}[Branch Constraint Set]
For branch $b$, let $\Gamma_b$ denote the set of branch constraints accumulated from selected disjunction choices during speculative expansion.
\end{definition}

\begin{definition}[Candidate Nogood]
A \emph{candidate nogood} is a set of class expressions $N \subseteq \Gamma_b$ obtained by greedy remove-one minimization from a contradictory branch $b$.
\end{definition}

\begin{definition}[Verified Nogood for Pruning]
A nogood $N$ is \emph{verified} if unsatisfiability is explicitly re-checked before insertion into the pruning set, i.e., $\mathcal{O} \cup N \models \bot$ is confirmed by the contradiction oracle used by the implementation.
\end{definition}

\begin{theorem}[Nogood Soundness for Pruning]\label{thm:nogood-soundness}
Let $N$ be a verified nogood admitted for pruning by SPACL. Then $N \models \bot$.
\end{theorem}

\begin{proof}
SPACL first obtains a contradictory branch ($\mathcal{O}\cup\Gamma_b \models \bot$), then greedily removes expressions while preserving contradiction, yielding a candidate $N$. A set is admitted as a pruning nogood only if contradiction is re-validated ($\mathcal{O}\cup N \models \bot$). Therefore every admitted pruning nogood is unsatisfiable, so $N \models \bot$.
\qed
\end{proof}

\begin{corollary}[Pruning Safety]\label{cor:pruning-safety}
If verified nogood $N$ subsumes branch constraints $\Gamma$ ($N \subseteq \Gamma$), then pruning the branch with constraints $\Gamma$ is sound.
\end{corollary}

\begin{proof}
From Theorem~\ref{thm:nogood-soundness}, $N \models \bot$.
Since $N \subseteq \Gamma$, $\Gamma$ contains all expressions of $N$.
Therefore $\Gamma \models \bot$ and the branch is unsatisfiable.
\qed
\end{proof}

\begin{remark}[Greedy Core Approximation]
SPACL uses greedy remove-one minimization rather than an exact minimum unsatisfiable subset (MUS) extraction. This preserves pruning soundness (Theorem~\ref{thm:nogood-soundness}) for verified nogoods but may still leave non-minimal cores; tighter MUS-style extraction remains future work.
\end{remark}

\subsection{Adaptive Threshold}
\label{sec:adaptive}

Problem complexity is estimated using:

\begin{equation}
\text{branches} = \max\left(1, \quad |\text{DetectedDisjunctions}| + 2 \times |\text{DisjointAxioms}| + \frac{|\text{Classes}|}{10}\right)
\label{eq:estimate}
\end{equation}

This heuristic accounts for:
\begin{itemize}
    \item \textbf{Detected disjunctions}: Direct signal for speculative branch creation
    \item \textbf{Disjointness axioms}: Each creates binary branching
    \item \textbf{Class hierarchy size}: Larger ontologies need more exploration
    \item \textbf{Conservative estimate}: Prefer sequential for uncertain cases
\end{itemize}

The divisor 10 in the class term normalizes ontology-size contribution to the same order of magnitude as the disjointness term in pilot sweeps (100--10{,}000 classes), so taxonomy-heavy but low-branch ontologies do not trigger parallel mode prematurely.

The threshold $\tau = 2000$ was selected from calibration sweeps that combined synthetic union/complement workloads with representative warm-up runs on DOID and GO-Basic (Section \ref{sec:threshold-tuning}). This conservative setting avoids pathological parallel activation on the 5k-disjunction synthetic stress case while preserving near-parity fallback on low-branch workloads.

\end{itemize}

\label{sec:components}

    \item Worker thread pool
    \item Nogood database ($\text{Arc}\langle\text{NogoodDatabase}\rangle$)
    \item Statistics collection
\end{itemize}

\textbf{ThreadLocalNogoodCache}: Per-worker cache with:
\begin{itemize}

\end{itemize}

\begin{itemize}

    \item LRU pruning when capacity exceeded (default: 10,000 nogoods)
    \item Hit count tracking for statistics
\end{itemize}

\subsection{Memory Management}
\label{sec:memory}

Memory usage scales with:
\begin{align}
\text{Memory} &= O(|\text{axioms}|) + O(|\text{workers}| \times |\text{depth}|) \\
&\quad + O(\min(|\text{nogoods}|, \text{max\_nogoods}))
\end{align}

Here, $|\text{nogoods}|$ is the number of learned conflicts generated during search, and $\text{max\_nogoods}$ is the configured retention cap of the global nogood store (default 10{,}000 in the current implementation). The $\min(\cdot)$ term reflects bounded retention: once the cap is reached, lower-utility entries are pruned.

Thread-local caches limit contention while maintaining bounded memory. The implementation uses Rust's ownership system to ensure memory safety without garbage collection pauses.

\subsection{Structural RDF/XML Parser and Materialization}
\label{sec:structural-parser}

The current large-ontology path uses a two-phase structural parser designed for handler-equivalent behavior on the implemented ALC/SHOIQ handling path by reusing the same semantic handlers as the standard parser.

\textbf{Phase A (Ingest + Intern)}:
\begin{itemize}
    \item Stream RDF/XML triples with Rio parser callbacks.
    \item Intern subject/predicate/object terms to dense integer IDs.
    \item Store compact records $\langle s\_id, p\_id, o\_id \rangle$ in arrival order.
\end{itemize}

\textbf{Phase B (Materialize)}:
\begin{itemize}
    \item Resolve IDs via dense caches and subject cursor fast-paths.
    \item Compute predicate tags.
    \item Dispatch each triple through \texttt{apply\_triple\_terms\_core} (same semantic path as baseline parser).
\end{itemize}

Algorithm~\ref{alg:structural-parser} gives the end-to-end structural ingest/materialize flow used in these experiments.

We model end-to-end load time as:
\begin{equation}
T_{\text{total}} = T_{\text{ingest}} + T_{\text{materialize}} + T_{\text{io}}
\label{eq:parser-total}
\end{equation}

For a cache layer $k \in \{\text{subj}, \text{pred}, \text{obj}\}$:
\begin{equation}
H_k = \frac{\text{hits}_k}{\text{hits}_k + \text{misses}_k}, \quad
E[t_k] = H_k \cdot t_{hit,k} + (1-H_k)\cdot t_{miss,k}
\label{eq:cache-hit}
\end{equation}

Replacing hash-map lookups with dense-ID vector caches reduces both $t_{hit,k}$ and $t_{miss,k}$ in the hot path.

To relate parser acceleration to overall runtime, we apply Amdahl's law:
\begin{equation}
S_{\text{e2e}} = \frac{1}{(1-f_{\text{parse}}) + \frac{f_{\text{parse}}}{S_{\text{parse}}}}
\label{eq:amdahl-parser}
\end{equation}
where $f_{\text{parse}}$ is the parse fraction before optimization and $S_{\text{parse}}$ is parser-stage speedup.

\begin{algorithm}[htbp]
\caption{Structural RDF/XML Parse with Handler-Equivalent Materialization}
\label{alg:structural-parser}
\begin{algorithmic}[1]
\Require RDF/XML stream $X$
\Ensure Ontology $\mathcal{O}$ targeting handler-equivalent behavior to the baseline parser on the implemented path
\State Initialize interner $I$, record buffer $R$, ontology $\mathcal{O}$
\For{each triple $t$ emitted by streaming RDF/XML parser}
    \State $s \gets \textsc{InternSubject}(I, t.subject)$
    \State $p \gets \textsc{InternPredicate}(I, t.predicate)$
    \State $o \gets \textsc{InternObject}(I, t.object)$
    \State Append $(s,p,o)$ to $R$
\EndFor
\State Initialize dense caches ($C_{subj}$, $C_{pred}$, $C_{obj}$)
\For{each record $(s,p,o)$ in $R$}
    \State $\hat{s} \gets \textsc{ResolveSubject}(s, C_{subj})$
    \State $\hat{p} \gets \textsc{ResolvePredicate}(p, C_{pred})$
    \State $\hat{o} \gets \textsc{ResolveObject}(o, C_{obj})$
    \State $\rho \gets \textsc{PredicateTag}(\hat{p})$
    \State \textsc{ApplyTripleTermsCore}$(\hat{s}, \rho, \hat{o}, \mathcal{O})$
\EndFor
\State \Return $\mathcal{O}$
\end{algorithmic}
\end{algorithm}

SPACL overhead vs sequential:

    \item \textbf{Medium ontologies} (1,000 classes): 0.99$\times$ (parity)

\subsection{Thread Pool Optimization (Phase 1)}
\label{sec:threadpool}

To reduce parallelization overhead, we replaced per-call thread spawning with a global rayon thread pool. This optimization eliminates the $\sim$200$\mu$s thread creation cost.

\textbf{Results}:
\begin{itemize}
    \item \textbf{Before}: 235$\mu$s overhead per call (thread spawn/teardown)
    \item \textbf{After}: 112$\mu$s overhead per call (thread pool reuse)
    \item \textbf{Improvement}: 2.1$\times$ faster for simple disjunctions
\end{itemize}

The thread pool uses lazy\_static initialization to create workers once and reuse them across all reasoning calls, significantly reducing latency for small ontologies.

\subsection{Adaptive Threshold Optimization (Phase 2)}
\label{sec:adaptive-opt}

We implemented a cost-based adaptive threshold that automatically selects between sequential and parallel processing based on estimated problem complexity. This avoids the $\sim$1ms parallel setup overhead for trivial work.

\textbf{Cost Model}:
\begin{align}
    \text{cost}(\text{ObjectUnionOf}(ops)) &= 50 \times |ops| + \sum \text{cost}(op) \\
    \text{cost}(\text{ObjectIntersectionOf}(ops)) &= 25 \times |ops| + \sum \text{cost}(op) \\
    \text{cost}(\text{ObjectComplementOf}(e)) &= 30 + \text{cost}(e) \\
    \text{cost}(\text{atomic}) &= 10
\end{align}

This gate is applied after the branch-count prefilter in Section \ref{sec:adaptive}: first, the disjunction-aware branch estimate with $\tau = 2000$ decides whether parallel exploration is considered at all; second, parallelism is enabled only when estimated cost $>$ 500$\mu$s $\times$ num\_workers.

\subsection{Threshold Tuning}
\label{sec:threshold-tuning}

We evaluated threshold settings for the two-stage adaptive policy. Table~\ref{tab:adaptive} reports the cost-gate behavior for small disjunctive cases, while Section~\ref{sec:adaptive} defines the branch-count prefilter.

We then ran explicit ablation sweeps over branch thresholds and cost coefficients using synthetic union/complement workloads plus representative warm-up real-world ontologies (DOID and GO-Basic). The exploratory sweep and conservative zoom-in sweep are archived under \path{results/history/adaptive_calibration_20260228_110137} and \path{results/history/adaptive_calibration_20260228_111338}; two final sensitivity reruns with forced-sequential baselines and one warm-up timing pass per mode are archived under \path{results/history/adaptive_calibration_20260228_235333} and \path{results/history/adaptive_calibration_20260228_235856}. Configurations were ranked by average regret against the best observed baseline wall-clock result on each sweep workload. In these traces, the disjunctive 5k stress case produced 1,000 detected disjunctions and an estimated branch count of 1,500; thresholds below that point triggered a severe always-parallel regression, while the robust regime consistently started at $\tau \ge 2000$. We therefore use these sweeps primarily to justify the safe operating region (`2000+`) of the branch gate rather than to claim a uniquely optimal default. The implementation keeps a conservative default branch-count threshold $\tau = 2000$, retains the 500$\mu$s cost gate, and applies the lighter 15$\mu$s nesting cost in the implementation defaults.

On the same calibration traces, DOID and GO-Basic stayed on the sequential path because the detector found no speculative disjunctions in those runs. This supports treating the adaptive gate as a workload-sensitive safety mechanism rather than as the primary explanation for the large biomedical wall-clock gains, which are discussed separately in Sections \ref{sec:realworld} and \ref{sec:competitor-benchmarks}.

\end{itemize}

\subsection{Nogood Learning Effectiveness}
\label{sec:nogood-eval}

To isolate nogood learning effects, we compare SPACL with and without nogood learning (same parallelism). Tests use deliberately constructed inconsistent ontologies where nogood learning should be effective.

\textbf{Observations}:
\begin{itemize}
    \item \textbf{Inconsistent ontologies}: 24\% speedup, 28\% branches pruned
    \item \textbf{Consistent ontologies}: Minimal overhead (2\%), nogoods rarely triggered

This would make branch-pruning gains directly comparable across workloads.

\subsection{Answering Research Questions}
\label{sec:answers}

\textbf{RQ1: Can speculative parallel tableaux with conflict-driven nogood learning improve consistency-checking performance?}  
\textbf{Answer}: Yes, for the evaluated prototype and workloads. In synthetic scalability tests, SPACL achieves up to 4.88$\times$ speedup at 10,000 classes, and nogood mechanisms reduce redundant branch exploration in inconsistent disjunctive workloads (Sections \ref{sec:scalability}, \ref{sec:nogood-eval}).

\textbf{RQ2: Can adaptive parallelism reduce overhead on small workloads while preserving scalability?}  
\textbf{Answer}: Yes. The adaptive strategy avoids unnecessary parallel execution on small cases (18--73$\times$ improvement versus always-parallel baselines) while retaining scalability on larger cases (Sections \ref{sec:adaptive}, \ref{sec:threshold-tuning}).

\textbf{RQ3: For large RDF/XML ontologies, which stages dominate runtime, and can structural parsing/materialization reduce the bottleneck while retaining implementation-aligned semantic behavior?}  
\textbf{Answer}: Parsing/materialization dominates large-ontology runtime in the baseline. The structural pipeline substantially reduces this cost while preserving handler-equivalent semantics for the implemented ALC/SHOIQ path, with stage-level A/B reductions reported in Section \ref{sec:realworld}, including a supplementary ChEBI cold-text stress pair that shows a similar trend.

\section{Conclusion and Future Work}
\label{sec:conclusion}

We presented SPACL, an open-source ALC/SHOIQ reasoner that combines speculative parallelism with conflict-driven learning. In the profile-mixed Standard Workload Suite (3 clean runs), SPACL has the lowest per-ontology median result on 4/7 ontologies and completes 21/21 cases, while ELK is faster on the two deepest hierarchy stress files (\texttt{hierarchy\_10000.owl}, \texttt{hierarchy\_100000.owl}) and Pellet is faster on \texttt{univ-bench.owl}. In the primary Large Head-to-Head benchmark on biomedical RDF/XML ontologies (doid/go-basic/uberon, 3 clean runs), the revised structural parser/materialization pipeline substantially improves end-to-end behavior, and SPACL has the lowest per-ontology median wall-clock time on all 3/3 tested ontologies among reasoners with successful comparable runs under a 900s timeout policy. A supplementary ChEBI cold-text stress pair shows a similar parser-dominant trend at a larger ontology scale without being folded into the main cross-reasoner large table. A suite-level failure-aware CEW view over comparable runs reports ELK as strongest on Standard and SPACL as strongest on Large. In the external OWL2Bench core-profile suite (DL/EL/QL/RL), three clean repeated runs show SPACL with 12/12 success and the lowest median wall-clock result on 4/4 ontologies among successful comparable runs in this harness. Head-to-head claims are reported as descriptive repeated-run outcomes for the stated benchmark harness.

\subsection{Summary of Contributions}
\label{sec:summary}

\begin{enumerate}
    \item Speculative work-stealing tableaux framework with conflict-driven nogood learning
    \item Adaptive overhead control via thresholding and global thread-pool execution
    \item Structural RDF/XML parsing/materialization with handler-equivalent semantics for the implemented ALC/SHOIQ path
    \item Reproducible containerized benchmarking and head-to-head evaluation protocol, including repeated Standard/Large/OWL2Bench validation
\end{enumerate}

\subsection{Limitations}
\label{sec:limitations}

\begin{enumerate}
    \item \textbf{Workload and Profile Differences}: Competitor comparison includes engines with different logic profiles (e.g., ELK for OWL EL++). Fastest wall-clock does not imply identical expressivity coverage.
    
    \item \textbf{Runtime Initialization Effects}: Standard-suite wall-clock comparisons include runtime startup and framework initialization costs, especially for JVM-based engines; these results should be interpreted as end-to-end deployment measurements rather than pure kernel timing.
    
    \item \textbf{Reproducibility Scope}: Results are measured under containerized CLI harness conditions (startup + parsing + reasoning). Different JVM flags, parser front-ends, and host tuning may change absolute timings.

    \item \textbf{Container Tag Drift}: Competitor containers are referenced by image family and run-time tags from benchmark dates. We record observed image digests for this revision (\texttt{image\_digests\_20260226.txt}), but strict digest pinning is not yet enforced by the benchmark harness.
    
    \item \textbf{Consistency Mix Across Suites}: The primary head-to-head suites use consistent ontologies, while inconsistent cases appear mainly in dedicated nogood ablations. Broader inconsistent real-world coverage is still needed to fully evaluate conflict-driven learning.
    
    \item \textbf{TBox-Only Evaluation}: Current benchmarks focus on classification (TBox). ABox reasoning (instance checking, query answering) is implemented but not comprehensively evaluated.
    
    \item \textbf{Single Machine}: SPACL targets single-node multi-core systems. Distributed reasoning across clusters is not supported.
    
    \item \textbf{Prototype Branch-Generation Scope}: The current speculative engine uses selected disjunction splitting with bounded speculative depth and timeout-based termination. Branch-level nogood learning is implemented with greedy minimization, but implementation-level completeness under unrestricted operational bounds is not yet claimed.
    
    \item \textbf{Deep-Hierarchy Performance Gap}: Robustness issues on extreme hierarchy chains were fixed in the current revision, but wall-clock on very deep taxonomy stress files is still behind EL-focused engines; additional parser/materialization optimization is needed for that regime.
    
    \item \textbf{Profile Support}: SPACL currently supports ALC/SHOIQ. Full OWL2 DL (with datatypes, keys) requires additional implementation.
\end{enumerate}

\subsection{Future Directions}
\label{sec:future}

\begin{enumerate}
    \item \textbf{GPU Acceleration}: Exploit massive parallelism for very large ontologies
    \item \textbf{Distributed Reasoning}: Extend to cluster environments
    \item \textbf{Incremental Reasoning}: Support evolving ontologies efficiently
    \item \textbf{ABox Optimization}: Apply speculative techniques to instance checking
    \item \textbf{Ontology Transaction Deployment}: Evaluate SPACL as a reasoning core for ontology transaction validation in distributed application workflows
    \item \textbf{Tool Integration}: Provide stable API/SDK interfaces for application-level integration
\end{enumerate}
