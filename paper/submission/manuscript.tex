\begin{abstract}
We present SPACL (Speculative Parallel Tableaux with Adaptive Conflict Learning), an open-source ALC/SHOIQ reasoner that combines speculative parallelism with conflict-driven learning. The reference implementation and repository are named \textit{Tableauxx}; throughout this manuscript, we use \textit{SPACL} for the algorithm/system and \textit{Tableauxx} only when referring to the software artifact. To our knowledge, SPACL is one of the few open implementations that integrates work-stealing parallelism with nogood learning in this setting, targeting exponential search behavior in tableau-based reasoning.

Our key contributions are: (1) a speculative parallel tableaux algorithm that explores branches concurrently with work-stealing; (2) an adaptive threshold mechanism that selects sequential or parallel execution based on estimated complexity, yielding 18--73$\times$ speedup over naive always-parallel execution on small ontologies; (3) thread-local nogood caching that reduces synchronization overhead by 80\%; (4) a global thread pool optimization that reduces overhead by 2.1$\times$; and (5) a Rust implementation with parser support for RDF/XML, OWL2 Functional, Manchester, and JSON-LD syntaxes. SPACL reaches $5\times$ speedup at 10,000 classes while keeping overhead below $2\times$ on small ontologies.

In the Small Workload Suite, SPACL records the best wall-clock time on 4/6 ontologies and is the only reasoner that completes all six cases in our containerized harness; \texttt{disjunctive\_simple.owl} and \texttt{hierarchy\_10000.owl} are faster on Pellet (with ELK also faster on \texttt{hierarchy\_10000.owl}). In the Large Head-to-Head benchmark (doid/go-basic/uberon), SPACL records the best wall-clock result on all three tested ontologies under an identical 900s timeout protocol. In an external OWL2Bench core-profile validation (DL/EL/QL/RL), three clean repeated runs show SPACL fastest by median wall-clock time on 4/4 ontologies with 12/12 successful runs in this harness. Timeout policy is reported explicitly in all benchmark tables (Small Workload Suite: 180s; Large Head-to-Head: 900s; OWL2Bench core profile suite: 300s). This improvement is driven by a structural RDF/XML parser and optimized materialization path that preserves the same ALC/SHOIQ semantic handlers as the baseline path; stage-level A/B runs show parse-only reductions of 96.96--97.98\% on doid/go-basic/uberon relative to the previous structural baseline.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Description Logics (DLs) provide the formal basis of the Semantic Web, and OWL2 DL is the corresponding W3C standard ontology language \cite{owl2}. Tableaux methods are still widely used for OWL2 DL reasoning because they provide sound and complete procedures for expressive TBox and ABox tasks \cite{dlhandbook}. The practical bottleneck is the combinatorial growth of tableau branches, especially on large ontologies. In this paper we target the ALC/SHOIQ fragment to study this bottleneck under expressive, disjunction-heavy workloads.

Modern hardware offers abundant parallel capacity, but OWL reasoners do not always convert that capacity into stable end-to-end gains. Some strong parallel systems are not openly developed (e.g., Konclude) \cite{konclude}, which limits reproducibility and extensibility. Many widely used open implementations (e.g., Pellet/Openllet and HermiT) remain primarily sequential in their core exploration loop \cite{pellet,openllet,hermit}. In addition, naive parallel branch exploration without conflict reuse repeats equivalent failing searches.
These constraints also motivate our longer-term goal: a reasoning core for ontology transaction validation in distributed, permissioned application settings. This paper focuses on the reasoning and parsing foundations required for that deployment path.

\subsection{Research Questions}
\label{sec:questions}

This work addresses the following research questions:

\begin{enumerate}[RQ1:]
    \item Can speculative parallel tableaux with conflict-driven nogood learning improve OWL2 DL consistency checking performance?
    \item Can adaptive parallelism reduce overhead on small workloads while preserving scalability on larger workloads?
    \item For large RDF/XML ontologies, which stages dominate end-to-end runtime, and how much can structural parsing/materialization optimization reduce this bottleneck while preserving OWL axiom fidelity?
\end{enumerate}

\subsection{Contributions}
\label{sec:contributions}

We address these questions through SPACL, with four main contributions:

\begin{enumerate}[C1:]
    \item \textbf{Speculative parallel tableaux with conflict learning}: SPACL combines work-stealing branch exploration with nogood extraction and reuse, reducing redundant failure exploration in non-deterministic search (Sections \ref{sec:workstealing}, \ref{sec:nogood}).
    
    \item \textbf{Adaptive overhead control}: We integrate threshold-based strategy selection with a global thread pool, reducing small-workload parallel overhead while preserving speedup at larger scales (Sections \ref{sec:adaptive}, \ref{sec:threadpool}).
    
    \item \textbf{Full-fidelity structural RDF/XML pipeline}: We implement a structural parsing/materialization path that preserves OWL axiom fidelity and substantially reduces parser-dominated runtime on large ontologies (Section \ref{sec:structural-parser}).
    
    \item \textbf{Reproducible end-to-end evaluation}: We provide a containerized benchmark protocol and artifacts for stage-level analysis and multi-reasoner head-to-head comparison (Sections \ref{sec:benchmark-protocol}, \ref{sec:competitor-benchmarks}).
\end{enumerate}

In Rust, these design choices translate into measurable gains: $5\times$ speedup at 10,000 classes, sub-microsecond per-class processing, and under $2\times$ overhead on small ontologies. On synthetic scalability workloads, SPACL reaches 26.2 million tableau operations per second.

\section{Related Work}
\label{sec:related}

This section summarizes prior work on tableau-based OWL reasoning, parallel and distributed execution, conflict-driven learning, and speculative execution. The goal is to position SPACL against the strongest established techniques used in practical reasoners.

\subsection{Tableau-Based Reasoning}

Tableau calculi remain a central approach for expressive description logic reasoning. Practical systems such as Pellet and HermiT demonstrated that sound and complete reasoning can be delivered at scale when rule ordering, blocking, and caching are engineered carefully~\cite{pellet,hermit,motik2009hypertableau}. In particular, the hypertableau line of work reduces non-deterministic branching pressure compared with classical tableau expansion, and individual-reuse techniques reduce model blow-up in challenging ontologies~\cite{motik2009hypertableau}.

State-of-the-art systems also combine calculi. Konclude integrates tableau-style reasoning with additional optimization layers and has shown strong performance on large benchmark sets~\cite{konclude,song2013complete,glimm2014coupling}. For tractable fragments, consequence-based reasoners such as ELK provide excellent throughput, illustrating how calculus design strongly influences scalability and implementation trade-offs~\cite{elk}.

\subsection{Parallel Reasoning}

Parallelization has been studied at multiple granularities. Shared-memory classification frameworks distribute satisfiability and subsumption checks across workers and report substantial speedup on multicore systems~\cite{quan2017parallel,quan2019framework}. Work-stealing style scheduling further improves load balancing when ontology subproblems are uneven~\cite{workstealing,quan2017parallel}.

Beyond classification, parallel ABox reasoning has explored dynamic splitting of model-construction tasks with cache reuse strategies to improve throughput on assertion-heavy workloads~\cite{steigmiller2020parallelised}. Distributed materialization frameworks (for less expressive profiles) use Spark-based execution and data-partitioned pipelines to scale to very large datasets~\cite{liu2017spowl,gu2015cichlid,benitez2023nora}. These results confirm the scalability potential of parallel infrastructures, while also highlighting that end-to-end performance is highly sensitive to reasoning profile, task type, and synchronization overhead.

\subsection{Conflict-Driven Learning}

Conflict-driven learning is a mature optimization paradigm in SAT/CSP, where learned clauses/nogoods prevent repeated exploration of failing search regions~\cite{marques1999grasp,glorian2020nacre}. In description logic systems, related ideas appear in more localized forms, such as nogood-aware individual reuse in hypertableau procedures~\cite{motik2009hypertableau}.

Hybrid reasoners that couple tableau and saturation procedures also demonstrate the value of reusing derived information to reduce repeated work during classification~\cite{song2013complete,glimm2014coupling,konclude}. However, openly documented integrations of branch-level speculative parallel tableau exploration with adaptive, reusable nogood learning remain limited in mainstream open-source OWL2 DL pipelines.

\subsection{Speculative Execution}

Speculative execution for OWL reasoning explores multiple non-deterministic branches concurrently and relies on cancellation/pruning once contradictions or witnesses are found. Work-stealing provides a practical runtime foundation for this model by balancing branch workloads dynamically~\cite{workstealing,quan2017parallel}. Lock-free and low-contention engineering techniques are also critical for obtaining net gains rather than synchronization-bound slowdowns~\cite{elk,quan2019framework}.

The primary systems challenge is overhead control. On easy workloads, naive always-parallel execution can underperform sequential reasoning because scheduling and coordination costs dominate. This motivates adaptive policies that switch between sequential and parallel execution based on estimated branch complexity~\cite{quan2017parallel}. A second challenge is conflict sharing: learned nogoods should be reused aggressively enough to prune search, but not so aggressively that contention erodes parallel benefit~\cite{steigmiller2020parallelised}.

\subsection{Summary and Positioning}

Prior literature provides strong building blocks: optimized tableau/hypertableau calculi~\cite{motik2009hypertableau,hermit}, high-throughput parallel classification frameworks~\cite{quan2017parallel,quan2019framework}, distributed materialization engines~\cite{liu2017spowl,gu2015cichlid,benitez2023nora}, and mature nogood machinery in constraint solving~\cite{glorian2020nacre}. SPACL is positioned at their intersection: branch-level speculative parallelism, adaptive overhead control, and practical nogood reuse designed for expressive OWL reasoning workloads.
To make this positioning analytically precise, the next section formalizes the decision model, branching behavior, and correctness arguments that underlie the implementation and evaluation.
\section{Formal Framework}
\label{sec:formal}

This section establishes the formal foundations of the SPACL algorithm, including the Description Logic syntax, tableaux calculus, and the mathematical basis for speculative parallelism with conflict learning.

\subsection{Description Logic ALC}
\label{sec:dl-syntax}

We operate in the Description Logic $\mathcal{ALC}$ (Attributive Language with Complement), which forms the basis of OWL2 DL. Our implementation extends to $\mathcal{ALCHOIQ}$ (ALC with role hierarchies, nominals, inverse roles, and qualified cardinality restrictions). The syntax is defined as follows:

\begin{definition}[ALC Syntax]
Let $\mathbf{C}$ be a set of concept names, $\mathbf{R}$ a set of role names, and $\mathbf{I}$ a set of individual names. \emph{Concepts} $C, D$ are defined by the grammar:
\begin{align}
C, D &::= A \mid \top \mid \bot \mid \neg C \mid C \sqcap D \mid C \sqcup D \mid \exists r.C \mid \forall r.C
\end{align}
where $A \in \mathbf{C}$ is an atomic concept and $r \in \mathbf{R}$ is a role.
\end{definition}

An \emph{ontology} $\mathcal{O} = (\mathcal{T}, \mathcal{A})$ consists of a TBox $\mathcal{T}$ (terminology) and ABox $\mathcal{A}$ (assertions):
\begin{align}
\mathcal{T} &::= \{C \sqsubseteq D\} \cup \{C \equiv D\} \\
\mathcal{A} &::= \{a : C\} \cup \{(a,b) : r\}
\end{align}

\subsection{Tableaux Calculus}
\label{sec:tableaux-calculus}

The tableaux algorithm determines consistency by constructing a \emph{completion graph} $\mathcal{G} = (V, E, \mathcal{L})$ where:
\begin{itemize}
    \item $V$ is a set of nodes (individuals)
    \item $E \subseteq V \times \mathbf{R} \times V$ are labeled edges (role assertions)
    \item $\mathcal{L}: V \rightarrow 2^{\mathcal{C}}$ maps nodes to sets of concepts
\end{itemize}

\begin{definition}[Clash]
A node $x \in V$ contains a \emph{clash} if either:
\begin{enumerate}
    \item $\{C, \neg C\} \subseteq \mathcal{L}(x)$ for some concept $C$, or
    \item $\bot \in \mathcal{L}(x)$
\end{enumerate}
\end{definition}

The tableaux algorithm applies \emph{expansion rules} until either a clash-free complete graph is found (consistent) or all branches lead to clashes (inconsistent).

\begin{table}[htbp]
\centering
\caption{Tableaux Expansion Rules for $\mathcal{ALC}$}
\label{tab:expansion-rules}
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Rule} & \textbf{Premise} & \textbf{Action} \\
\midrule
$(\sqcap)$ & $x : C \sqcap D \in \mathcal{L}(x)$ & $\mathcal{L}(x) := \mathcal{L}(x) \cup \{C, D\}$ \\
$(\sqcup)$ & $x : C \sqcup D \in \mathcal{L}(x)$ & Create branches: $\mathcal{L}_1(x) = \mathcal{L}(x) \cup \{C\}$, \\
& & $\mathcal{L}_2(x) = \mathcal{L}(x) \cup \{D\}$ \\
$(\exists)$ & $x : \exists r.C \in \mathcal{L}(x)$ & Add new $y$, $(x,y):r$, $y:C$ \\
$(\forall)$ & $x : \forall r.C \in \mathcal{L}(x)$, $(x,y):r$ & $\mathcal{L}(y) := \mathcal{L}(y) \cup \{C\}$ \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Speculative Parallelism Model}
\label{sec:parallel-model}

Traditional tableaux processing applies the $(\sqcup)$-rule sequentially, creating a binary decision tree with depth equal to the number of disjunctions. SPACL introduces \emph{speculative parallelism}:

\begin{definition}[Speculative Branch]
Given a disjunction $C \sqcup D$ at node $x$, a \emph{speculative branch} is a pair of subgraphs $(\mathcal{G}_1, \mathcal{G}_2)$ where:
\begin{align}
\mathcal{G}_1 &= (V, E, \mathcal{L}[x \mapsto \mathcal{L}(x) \cup \{C\}]) \\
\mathcal{G}_2 &= (V, E, \mathcal{L}[x \mapsto \mathcal{L}(x) \cup \{D\}])
\end{align}
Both branches are explored concurrently by different workers.
\end{definition}

\begin{theorem}[Parallel Speedup Bound]
Let $T_{seq}$ be the sequential tableaux runtime and $T_{par}$ the speculative parallel runtime with $p$ workers. For an ontology with $d$ disjunctions:
\begin{align}
T_{par} \geq \frac{T_{seq}}{p} \cdot (1 + \alpha \cdot o_{sync})
\end{align}
where $\alpha$ is the fraction of work requiring synchronization and $o_{sync}$ is the synchronization overhead.
\end{theorem}

\subsection{Nogood Learning}
\label{sec:nogood-formal}

\begin{definition}[Nogood]
A \emph{nogood} is a set of class expressions $\mathcal{N} = \{C_1, C_2, \ldots, C_n\}$ such that:
\begin{align}
\bigcap_{i=1}^{n} C_i \sqsubseteq \bot
\end{align}
i.e., the conjunction of all expressions in $\mathcal{N}$ is unsatisfiable.
\end{definition}

\begin{definition}[Nogood Subsumption]
A nogood $\mathcal{N}$ \emph{subsumes} a branch with constraints $\Gamma$ if:
\begin{align}
\mathcal{N} \subseteq \Gamma
\end{align}
If $\mathcal{N}$ subsumes $\Gamma$, the branch can be immediately pruned.
\end{definition}

\begin{algorithm}[htbp]
\caption{Nogood Extraction}
\label{alg:nogood}
\begin{algorithmic}[1]
\Require Clashed node $x$ with label $\mathcal{L}(x)$
\Ensure Nogood $\mathcal{N}$
\State $\mathcal{N} \gets \emptyset$
\For{each $C \in \mathcal{L}(x)$}
    \If{$C$ is non-atomic}
        \State $\mathcal{N} \gets \mathcal{N} \cup \{C\}$
    \EndIf
\EndFor
\State \Return $\textsc{Minimize}(\mathcal{N})$ \Comment{Remove redundant elements}
\end{algorithmic}
\end{algorithm}

\subsection{Adaptive Threshold}
\label{sec:adaptive-formal}

The adaptive threshold mechanism uses cost estimation to decide between sequential and parallel processing.

\begin{definition}[Disjunction Cost]
The cost of a disjunction $C = C_1 \sqcup C_2 \sqcup \cdots \sqcup C_n$ is:
\begin{align}
\text{cost}(C) = c_{base} + \sum_{i=1}^{n} \text{cost}(C_i)
\end{align}
where $c_{base}$ is the base processing cost per disjunction.
\end{definition}

\begin{definition}[Parallelization Threshold]
The adaptive threshold $\tau$ for ontology $\mathcal{O}$ with disjunctions $\mathcal{D}$ is:
\begin{align}
\tau(\mathcal{O}) = \beta \cdot \sum_{C \in \mathcal{D}} \text{cost}(C) \cdot \log(1 + |\mathcal{O}|)
\end{align}
where $\beta$ is a tuning parameter and $|\mathcal{O}|$ is the ontology size.
\end{definition}
In practice, $\beta$ is chosen to map symbolic disjunction-cost scores to observed scheduling overhead on the target hardware. We initialize $\beta$ from pilot microbenchmarks (sequential vs always-parallel crossover), then keep it fixed for all reported suites rather than retuning per ontology.

\begin{theorem}[Adaptive Optimality]
For an ontology $\mathcal{O}$ with total disjunction cost $D$, let $T_{seq}$ be sequential time and $T_{par}$ parallel time with overhead $o_{par}$. The adaptive mechanism achieves speedup when:
\begin{align}
D > \frac{p \cdot o_{par}}{1 - \alpha}
\end{align}
where $p$ is the number of workers and $\alpha$ is the nogood hit rate.
\end{theorem}

\subsection{Correctness}
\label{sec:correctness}

\begin{theorem}[Soundness]
If SPACL returns \emph{consistent} for ontology $\mathcal{O}$, then $\mathcal{O}$ is satisfiable.
\end{theorem}

\begin{proof}
SPACL explores branches speculatively but applies the same expansion rules as sequential tableaux. Each worker maintains a valid completion graph. If any worker finds a clash-free graph, that graph represents a model of $\mathcal{O}$. The early termination when one branch succeeds preserves the existence of a model.
\end{proof}

\begin{theorem}[Completeness]
If ontology $\mathcal{O}$ is consistent, SPACL will eventually return \emph{consistent}.
\end{theorem}

\begin{proof}
Speculative parallelism explores all branches of the search space (possibly concurrently). Nogood pruning only removes branches that are proven unsatisfiable. Since the sequential tableaux algorithm is complete and SPACL explores a superset of the sequential execution paths, it will find a model if one exists.
\end{proof}

\begin{enumerate}
    \item \textbf{Speculative Work-Stealing}: Branches are dynamically distributed across worker threads using a work-stealing deque \cite{workstealing}. Each worker maintains a local queue, stealing from others when idle.
    
    \item \textbf{Nogood Learning}: Contradictions are analyzed to extract minimal unsatisfiable assertion sets (nogoods). These are cached and checked before branch expansion.
    
    \item \textbf{Adaptive Thresholding}: Problem complexity is estimated from axiom structure. Problems below a threshold use sequential processing; larger problems benefit from parallelism.
\end{enumerate}

Algorithm \ref{alg:spacl} presents the main consistency checking procedure.

\begin{algorithm}[htbp]
\caption{SPACL Consistency Check}
\label{alg:spacl}
\begin{algorithmic}[1]
\Require Ontology $O$, Configuration $C$
\Ensure \textbf{true} if $O$ is consistent, \textbf{false} otherwise

\State $branches \gets \textsc{EstimateBranchCount}(O)$
\If{$branches < C.\text{parallel\_threshold}$} \label{line:threshold}
    \State \Return $\textsc{SequentialConsistencyCheck}(O)$ \label{line:sequential}
\EndIf

\State $Q \gets \textsc{InitializeWorkQueue}(O)$ \label{line:init}
\State $N \gets \textsc{InitializeNogoodDatabase}()$
\State $W[1..C.\text{num\_workers}] \gets \textsc{StartWorkers}(Q, N, C)$

\While{$\textsc{NotTerminated}()$}
    \State $result \gets \textsc{CollectResults}(C.\text{timeout})$
    \If{$result = \text{SAT}$} \Return \textbf{true}
    \ElsIf{$result = \text{UNSAT}$} \Return \textbf{false}
    \ElsIf{$result = \text{TIMEOUT}$} \Return \textbf{UNKNOWN}
    \EndIf
\EndWhile
\end{algorithmic}
\end{algorithm}

\subsection{Speculative Work-Stealing}
\label{sec:workstealing}

The work-stealing scheduler provides dynamic load balancing without centralized coordination. Each worker thread maintains:

\begin{itemize}
    \item \textbf{Local Queue}: Double-ended queue (deque) of pending WorkItems
    \item \textbf{Stealer Handle}: Allows other threads to steal work from the tail
    \item \textbf{Local Statistics}: Branch counts, nogood hits, cache metrics
\end{itemize}

When a worker encounters a disjunction ($A \sqcup B$), it creates WorkItems for both branches and pushes them to its local queue. Idle workers steal from the tail of busy workers' queues using a lock-free deque-based work-stealing strategy \cite{workstealing}.

\subsection{Nogood Learning and Caching}
\label{sec:nogood}

A nogood is a set of class expressions that cannot be simultaneously satisfied.

\begin{definition}[Nogood]
\label{def:nogood}
A nogood is a set of class assertions $\{C_i(a)\}_{i=1}^{n}$ such that their conjunction is unsatisfiable:
\begin{equation}
\bigsqcap_{i=1}^{n} C_i \sqsubseteq \bot
\end{equation}
\end{definition}

When a contradiction is detected, SPACL extracts the minimal assertion set causing the clash and stores it as a nogood. Before expanding a node, its assertion set is checked against known nogoods.

\subsubsection{Thread-Local Caching}
\label{sec:cache}

Each worker maintains three levels of nogood storage:

\begin{enumerate}
    \item \textbf{Local Nogoods}: Discovered by this worker (no locks needed)
    \item \textbf{Cached Global Nogoods}: Copy of shared database (periodically synced)
    \item \textbf{Global Database}: Shared across all workers (synchronized access)
\end{enumerate}

The cache check algorithm proceeds as follows:

\begin{enumerate}
    \item Check local nogoods (fastest, no synchronization)
    \item Check cached global nogoods (fast, no synchronization)
    \item If cache miss and sync interval reached:
    \begin{enumerate}
        \item Acquire read lock on global database
        \item Update cache with new nogoods
        \item Release lock
        \item Re-check with updated cache
    \end{enumerate}
    \item If still no hit, proceed with expansion
\end{enumerate}

Synchronization occurs every $N$ checks (default: 100), amortizing lock overhead.

\subsubsection{Correctness of Nogood Learning}
\label{sec:nogood-correctness}

We establish the soundness of SPACL's nogood learning mechanism.

\begin{definition}[Test Expression Set]
Let $\mathcal{O}$ be an ontology with SubClassOf axioms $\mathcal{A} = \{C_i \sqsubseteq D_i\}$. The \emph{test expression set} $T(\mathcal{O})$ is:
\begin{equation}
T(\mathcal{O}) = \{C_i \mid C_i \sqsubseteq D_i \in \mathcal{A}\} \cup \{D_i \mid C_i \sqsubseteq D_i \in \mathcal{A}\}
\end{equation}
\end{definition}

\begin{definition}[Nogood]
A \emph{nogood} is a set of class expressions $N \subseteq T(\mathcal{O})$ such that $N \models \bot$ (unsatisfiable).
\end{definition}

\begin{theorem}[Nogood Soundness]\label{thm:nogood-soundness}
Let $N$ be a set learned by SPACL from a contradictory branch. Then $N \models \bot$.
\end{theorem}

\begin{proof}
When SPACL detects a contradiction on a branch with assertions $A$, it creates nogood $N = T(\mathcal{O})$.

By the tableau expansion rules applied to $A$, we have $A \models \bot$.
Since $A \subseteq T(\mathcal{O}) = N$ and tableau reasoning is monotonic, $N \models \bot$.
\qed
\end{proof}

\begin{corollary}[Pruning Safety]\label{cor:pruning-safety}
If nogood $N$ subsumes branch assertions $B$ ($N \subseteq B$), then pruning $B$ is sound.
\end{corollary}

\begin{proof}
From Theorem~\ref{thm:nogood-soundness}, $N \models \bot$.
Since $N \subseteq B$, $B$ contains all assertions of $N$.
Therefore $B \models \bot$ and the branch is unsatisfiable.
\qed
\end{proof}

\begin{remark}[Conservative Over-Approximation]
SPACL uses $N = T(\mathcal{O})$ rather than the minimal unsatisfiable core of $A$. While this is sound (Theorem~\ref{thm:nogood-soundness}), it may miss pruning opportunities. Minimal nogood extraction is left as future work.
\end{remark}

\subsection{Adaptive Threshold}
\label{sec:adaptive}

Problem complexity is estimated using:

\begin{equation}
\text{branches} = \max\left(1, \quad 2 \times |\text{DisjointAxioms}| + \frac{|\text{Classes}|}{10}\right)
\label{eq:estimate}
\end{equation}

This heuristic accounts for:
\begin{itemize}
    \item \textbf{Disjointness axioms}: Each creates binary branching
    \item \textbf{Class hierarchy size}: Larger ontologies need more exploration
    \item \textbf{Conservative estimate}: Prefer sequential for uncertain cases
\end{itemize}

The divisor 10 in the class term normalizes ontology-size contribution to the same order of magnitude as the disjointness term in pilot sweeps (100--10{,}000 classes), so taxonomy-heavy but low-branch ontologies do not trigger parallel mode prematurely.

The threshold $\tau = 100$ was empirically determined through benchmark analysis (Section \ref{sec:threshold-tuning}).

\subsection{Core Components}
\label{sec:components}

\textbf{SpeculativeTableauxReasoner}: Main entry point managing:
\begin{itemize}
    \item Ontology reference ($\text{Arc}\langle\text{Ontology}\rangle$)
    \item Worker thread pool
    \item Nogood database ($\text{Arc}\langle\text{NogoodDatabase}\rangle$)
    \item Statistics collection
\end{itemize}

\textbf{ThreadLocalNogoodCache}: Per-worker cache with:

\begin{itemize}

    \item Hit count tracking for statistics
\end{itemize}

\subsection{Memory Management}
\label{sec:memory}

Memory usage scales with:
\begin{align}
\text{Memory} &= O(|\text{axioms}|) + O(|\text{workers}| \times |\text{depth}|) \\
&\quad + O(\min(|\text{nogoods}|, \text{max\_nogoods}))
\end{align}

Here, $|\text{nogoods}|$ is the number of learned conflicts generated during search, and $\text{max\_nogoods}$ is the configured retention cap of the global nogood store (default 10{,}000 in the current implementation). The $\min(\cdot)$ term reflects bounded retention: once the cap is reached, lower-utility entries are pruned.

Thread-local caches limit contention while maintaining bounded memory. The implementation uses Rust's ownership system to ensure memory safety without garbage collection pauses.

\subsection{Structural RDF/XML Parser and Materialization}
\label{sec:structural-parser}

The current large-ontology path uses a two-phase structural parser that preserves semantic equivalence for the implemented ALC/SHOIQ handling path by reusing the same semantic handlers as the standard parser.

\textbf{Phase A (Ingest + Intern)}:
\begin{itemize}
    \item Stream RDF/XML triples with Rio parser callbacks.
    \item Intern subject/predicate/object terms to dense integer IDs.
    \item Store compact records $\langle s\_id, p\_id, o\_id \rangle$ in arrival order.
\end{itemize}

\textbf{Phase B (Materialize)}:
\begin{itemize}
    \item Resolve IDs via dense caches and subject cursor fast-paths.
    \item Compute predicate tags.
    \item Dispatch each triple through \texttt{apply\_triple\_terms\_core} (same semantic path as baseline parser).
\end{itemize}

We model end-to-end load time as:
\begin{equation}
T_{\text{total}} = T_{\text{ingest}} + T_{\text{materialize}} + T_{\text{io}}
\label{eq:parser-total}
\end{equation}

For a cache layer $k \in \{\text{subj}, \text{pred}, \text{obj}\}$:
\begin{equation}
H_k = \frac{\text{hits}_k}{\text{hits}_k + \text{misses}_k}, \quad
E[t_k] = H_k \cdot t_{hit,k} + (1-H_k)\cdot t_{miss,k}
\label{eq:cache-hit}
\end{equation}

Replacing hash-map lookups with dense-ID vector caches reduces both $t_{hit,k}$ and $t_{miss,k}$ in the hot path.

To relate parser acceleration to overall runtime, we apply Amdahl's law:
\begin{equation}
S_{\text{e2e}} = \frac{1}{(1-f_{\text{parse}}) + \frac{f_{\text{parse}}}{S_{\text{parse}}}}
\label{eq:amdahl-parser}
\end{equation}
where $f_{\text{parse}}$ is the parse fraction before optimization and $S_{\text{parse}}$ is parser-stage speedup.

\begin{algorithm}[htbp]
\caption{Structural RDF/XML Parse with Full-Fidelity Materialization}
\label{alg:structural-parser}
\begin{algorithmic}[1]
\Require RDF/XML stream $X$
\Ensure Ontology $\mathcal{O}$ with full semantic equivalence to baseline parser
\State Initialize interner $I$, record buffer $R$, ontology $\mathcal{O}$
\For{each triple $t$ emitted by streaming RDF/XML parser}
    \State $s \gets \textsc{InternSubject}(I, t.subject)$
    \State $p \gets \textsc{InternPredicate}(I, t.predicate)$
    \State $o \gets \textsc{InternObject}(I, t.object)$
    \State Append $(s,p,o)$ to $R$
\EndFor
\State Initialize dense caches ($C_{subj}$, $C_{pred}$, $C_{obj}$)
\For{each record $(s,p,o)$ in $R$}
    \State $\hat{s} \gets \textsc{ResolveSubject}(s, C_{subj})$
    \State $\hat{p} \gets \textsc{ResolvePredicate}(p, C_{pred})$
    \State $\hat{o} \gets \textsc{ResolveObject}(o, C_{obj})$
    \State $\tau \gets \textsc{PredicateTag}(\hat{p})$
    \State \textsc{ApplyTripleTermsCore}$(\hat{s}, \tau, \hat{o}, \mathcal{O})$
\EndFor
\State \Return $\mathcal{O}$
\end{algorithmic}
\end{algorithm}

\textbf{Results}:
\begin{itemize}
    \item \textbf{Before}: 235$\mu$s overhead per call (thread spawn/teardown)
    \item \textbf{After}: 112$\mu$s overhead per call (thread pool reuse)
    \item \textbf{Improvement}: 2.1$\times$ faster for simple disjunctions
\end{itemize}

The thread pool uses lazy\_static initialization to create workers once and reuse them across all reasoning calls, significantly reducing latency for small ontologies.

\subsection{Adaptive Threshold Optimization (Phase 2)}
\label{sec:adaptive-opt}

We implemented a cost-based adaptive threshold that automatically selects between sequential and parallel processing based on estimated problem complexity. This avoids the $\sim$1ms parallel setup overhead for trivial work.

\textbf{Cost Model}:
\begin{align}
    \text{cost}(\text{ObjectUnionOf}(ops)) &= 50 \times |ops| + \sum \text{cost}(op) \\
    \text{cost}(\text{ObjectIntersectionOf}(ops)) &= 25 \times |ops| + \sum \text{cost}(op) \\
    \text{cost}(\text{ObjectComplementOf}(e)) &= 30 + \text{cost}(e) \\
    \text{cost}(\text{atomic}) &= 10
\end{align}

This gate is applied after the branch-count prefilter in Section \ref{sec:adaptive}: first, branching estimate $\tau = 100$ decides whether parallel exploration is considered at all; second, parallelism is enabled only when estimated cost $>$ 500$\mu$s $\times$ num\_workers.

\subsection{Analysis: JVM Startup Overhead}
\label{sec:jvm-overhead}

Small-workload wall-clock numbers should be interpreted as end-to-end deployment measurements rather than pure reasoning-kernel speed. In the Small Workload Suite, most JVM-based engines cluster around 2.7--4.3 seconds wall time on successful runs, indicating substantial fixed runtime initialization costs. SPACL (Tableauxx implementation) completes all six small cases; four cases are near 2.0--2.2 seconds, \texttt{disjunctive\_simple.owl} is 3.82 seconds, and \texttt{hierarchy\_10000.owl} is 8.27 seconds due to parser/materialization overhead in the text-loading path.

\textbf{Takeaway}: Wall-clock comparisons in this section reflect startup + parsing + reasoning together. They are useful for practical CLI deployment analysis, but should not be interpreted as parser-agnostic or reasoning-kernel-only measurements.

\subsection{Threshold Tuning}
\label{sec:threshold-tuning}

We evaluated threshold settings for the two-stage adaptive policy. Table~\ref{tab:adaptive} reports the cost-gate behavior for small disjunctive cases, while Section~\ref{sec:adaptive} defines the branch-count prefilter.

The default branch-count threshold $\tau = 100$ provided stable crossover behavior in our synthetic scaling tests (around 1,000 classes in Table~\ref{tab:scalability}); the cost gate ($>$ 500$\mu$s $\times$ workers) then avoids trivial parallel work inside that regime.

\subsection{Answering Research Questions}
\label{sec:answers}

\textbf{RQ1: Can speculative parallel tableaux with conflict-driven nogood learning improve consistency-checking performance?}  
\textbf{Answer}: Yes. SPACL achieves up to 4.88$\times$ speedup at 10,000 classes in scalability tests, and nogood mechanisms reduce redundant branch exploration in inconsistent disjunctive workloads (Sections \ref{sec:scalability}, \ref{sec:nogood-eval}).

\textbf{RQ2: Can adaptive parallelism reduce overhead on small workloads while preserving scalability?}  
\textbf{Answer}: Yes. The adaptive strategy avoids unnecessary parallel execution on small cases (18--73$\times$ improvement versus always-parallel baselines) while retaining scalability on larger cases (Sections \ref{sec:adaptive}, \ref{sec:threshold-tuning}).

\textbf{RQ3: For large RDF/XML ontologies, which stages dominate runtime, and can structural parsing/materialization reduce the bottleneck with full fidelity?}  
\textbf{Answer}: Parsing/materialization dominates large-ontology runtime in the baseline. The structural pipeline substantially reduces this cost while preserving handler-equivalent semantics for the implemented ALC/SHOIQ path, with stage-level A/B reductions reported in Section \ref{sec:realworld}.

\section{Conclusion and Future Work}
\label{sec:conclusion}

We presented SPACL, an open-source ALC/SHOIQ reasoner that combines speculative parallelism with conflict-driven learning. In the Small Workload Suite, SPACL achieves the best wall-clock result on 4/6 ontologies and completes 6/6 runs in our harness; \texttt{disjunctive\_simple.owl} and \texttt{hierarchy\_10000.owl} are faster on Pellet (with ELK also faster on \texttt{hierarchy\_10000.owl}). In the Large Head-to-Head benchmark on biomedical RDF/XML ontologies (doid/go-basic/uberon), the revised structural parser/materialization pipeline substantially improves end-to-end behavior, and SPACL records the best wall-clock time on all 3/3 tested ontologies under a 900s timeout policy. In the external OWL2Bench core-profile suite (DL/EL/QL/RL), three clean repeated runs show SPACL with 12/12 success and the best median wall-clock result on 4/4 ontologies in this harness.

\subsection{Summary of Contributions}
\label{sec:summary}

\begin{enumerate}
    \item Speculative work-stealing tableaux with conflict-driven nogood learning
    \item Adaptive overhead control via thresholding and global thread-pool execution
    \item Structural RDF/XML parsing/materialization with handler-equivalent semantics for the implemented ALC/SHOIQ path
    \item Reproducible containerized benchmarking and head-to-head evaluation protocol, including repeated OWL2Bench core-profile validation
\end{enumerate}

\subsection{Limitations}
\label{sec:limitations}

\begin{enumerate}
    \item \textbf{Workload and Profile Differences}: Competitor comparison includes engines with different logic profiles (e.g., ELK for OWL EL++). Fastest wall-clock does not imply identical expressivity coverage.
    
    \item \textbf{Runtime Initialization Effects}: Small-workload wall-clock comparisons include runtime startup and framework initialization costs, especially for JVM-based engines; these results should be interpreted as end-to-end deployment measurements rather than pure kernel timing.
    
    \item \textbf{Reproducibility Scope}: Results are measured under containerized CLI harness conditions (startup + parsing + reasoning). Different JVM flags, parser front-ends, and host tuning may change absolute timings.
    
    \item \textbf{Container Tag Drift}: Competitor containers are referenced by image family and run-time tags from benchmark dates; digest-pinned images are not yet enforced in this manuscript pipeline.
    
    \item \textbf{Consistency Mix Across Suites}: The primary head-to-head suites use consistent ontologies, while inconsistent cases appear mainly in dedicated nogood ablations. Broader inconsistent real-world coverage is still needed to fully evaluate conflict-driven learning.
    
    \item \textbf{TBox-Only Evaluation}: Current benchmarks focus on classification (TBox). ABox reasoning (instance checking, query answering) is implemented but not comprehensively evaluated.
    
    \item \textbf{Single Machine}: SPACL targets single-node multi-core systems. Distributed reasoning across clusters is not supported.
    
    \item \textbf{Profile Support}: SPACL currently supports ALC/SHOIQ. Full OWL2 DL (with datatypes, keys) requires additional implementation.
\end{enumerate}

\subsection{Future Directions}
\label{sec:future}

\begin{enumerate}
    \item \textbf{GPU Acceleration}: Exploit massive parallelism for very large ontologies
    \item \textbf{Distributed Reasoning}: Extend to cluster environments
    \item \textbf{Incremental Reasoning}: Support evolving ontologies efficiently
    \item \textbf{ABox Optimization}: Apply speculative techniques to instance checking
    \item \textbf{Ontology Transaction Deployment}: Evaluate SPACL as a reasoning core for ontology transaction validation in distributed application workflows
    \item \textbf{Tool Integration}: Provide stable API/SDK interfaces for application-level integration
\end{enumerate}
